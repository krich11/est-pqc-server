# Crypto Architecture

## Cryptographic Baseline
The bootstrap is configured to use the system-provided OpenSSL 3 library only. No vendored OpenSSL and no alternate crypto providers are used.

## System OpenSSL Binding Strategy
- Rust crate: `openssl = { version = "0.10", default-features = false }`
- Build enforcement:
  - `build.rs` checks for OpenSSL 3.x
  - `build.rs` emits:
    - `cargo:rustc-link-lib=ssl`
    - `cargo:rustc-link-lib=crypto`
- Build environment used for validation:
  - `OPENSSL_DIR=/opt/homebrew/opt/openssl@3.5`
  - `OPENSSL_LIB_DIR=/opt/homebrew/opt/openssl@3.5/lib`
  - `OPENSSL_INCLUDE_DIR=/opt/homebrew/opt/openssl@3.5/include`
  - `PKG_CONFIG_PATH=/opt/homebrew/opt/openssl@3.5/lib/pkgconfig`

## Detected Issues
- OpenSSL FIPS provider is not currently available
- Current platform is `macos`; RFC 7030 EST server testing, QA, and RPM packaging must run on the Linux host `192.168.200.120` throughout development

## Fix Recommendations Logged During Bootstrap
- Install the OpenSSL FIPS provider package on Linux and configure `OPENSSL_CONF` to load the `fips` provider
- On RHEL/Fedora: run `sudo fips-mode-setup --enable`
- On Debian/Ubuntu: install `openssl-fips-provider` if available and configure `/etc/ssl/openssl.cnf`
- Use the current generated configuration for local editing and compile checks only
- Run the EST server under test on the Linux host `192.168.200.120` for all RFC 7030 validation and QA stages
- Run the test client on the local development system while targeting the Linux-hosted EST server on `192.168.200.120`
- Export `OPENSSL_DIR=/opt/homebrew/opt/openssl@3.5` and `PKG_CONFIG_PATH=/opt/homebrew/opt/openssl@3.5/lib/pkgconfig` before local development builds

## Current Architectural Decision
Bootstrap is complete with a strict system-OpenSSL path and verified PQ capability detection. FIPS enablement remains deferred until execution on a Linux system with an installed FIPS provider. The authoritative EST QA topology is now fixed: the EST server runs on Linux host `192.168.200.120`, while the test client runs on the local development system and performs remote RFC 7030 validation against that Linux host during every stage of development.

## Linux-Hosted Validation Adjustments
- The Linux host uses system OpenSSL `3.0.2`, so remote EST runtime must not depend on macOS-specific configured absolute OpenSSL paths
- EST runtime OpenSSL detection was corrected to fall back to the system `openssl` binary when configured absolute paths are missing on the active host
- Remote QA restart handling was corrected to replace stale deleted EST listener processes before validation
- Final Linux-hosted EST validation passed cleanly against `https://192.168.200.120:8443`
